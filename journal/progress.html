1 What We Are Building

VELA — Automated M-Pesa Payment Collection for Small Businesses

Vela is a B2B SaaS platform that removes the manual work of chasing M-Pesa payments. Target customers are cloud kitchens, landlords, schools, gyms, and small businesses in Kenya who currently rely on M-Pesa screenshots and manual reconciliation.

  • A business signs up to Vela and provides their M-Pesa shortcode details.
  • Their customers receive an automatic payment prompt on their phone (STK Push).
  • When the customer pays, Vela receives a real-time callback from Safaricom and logs the payment automatically.
  • No screenshots. No manual chasing. No guessing.
The 90-day plan: Build the core payment engine in days 1–30, acquire 3 paying merchants in days 31–60, then refine and scale in days 61–90.

2 Python Virtual Environment

Why we use a virtual environment

Python packages installed globally conflict when different projects need different versions of the same library. A virtual environment is an isolated folder (venv/) that contains its own Python interpreter and its own set of installed packages — completely separate from everything else on your machine.

Why this matters for Vela: FastAPI, httpx, supabase-py, and python-dotenv are all installed inside venv/. If you pip-install them globally, they could clash with another project. Isolation keeps the build clean and reproducible.

Creating the virtual environment (one time only)

python -m venv venv

This creates a folder called venv/ in your project root. Inside it is a full copy of the Python interpreter and a Scripts/ folder with its own pip.

Activating it manually (when terminal does not activate it automatically)

# PowerShell (VS Code default)
.\venv\Scripts\Activate.ps1

# If PowerShell blocks scripts, run this once first:
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser

When activated, your terminal prompt changes to show (venv) at the start. That is the signal that you are inside the isolated environment.

Why it seems to activate itself in new terminals

VS Code has a setting: Python: Terminal Activate Environment. When it is turned on, VS Code detects the venv/ folder in your project and automatically runs Activate.ps1 every time a new terminal opens inside that workspace.

This is not magic — VS Code is doing it for you. It reads the Python interpreter you selected (bottom status bar), sees it points to venv/Scripts/python.exe, and activates the environment before you type anything. If it ever stops doing this, just activate manually using the command above.

Key rule: Always confirm (venv) is in your prompt before running pip install or uvicorn. If it is not there, your packages go to the wrong place.

3 Safaricom Daraja Developer Portal

What Daraja is

Daraja is Safaricom's API gateway — the technical bridge between software systems and M-Pesa. Without Daraja, there is no way to programmatically trigger or receive M-Pesa payments. All STK Push requests, callbacks, and B2C payments go through Daraja.

Account strategy: one developer account, many apps

Vela uses one Safaricom Developer Portal account (owned by Fredrick / Vela), and creates a separate App inside it for each merchant in production. This means:

  • Vela manages one login, one developer relationship with Safaricom.
  • Each merchant gets their own Consumer Key + Consumer Secret tied to their own M-Pesa shortcode.
  • Credentials are isolated — one merchant's credentials cannot touch another's transactions.
Why not one app for all merchants? A single app would mean one shortcode, and M-Pesa money would land in one account. Each merchant needs their own till/paybill number, so each needs their own Daraja app linked to that shortcode.

Sandbox credentials obtained

The sandbox is a test environment — no real money moves. We created an app called VEla in the sandbox and retrieved:

Why this approach

If credentials were written directly in Python files and committed to git, anyone with access to the repository would have full access to your M-Pesa account and database. The .env file is listed in .gitignore, so git never tracks it. python-dotenv reads it at runtime and injects the values as environment variables that Python code can access with os.getenv().

Rule: If someone asks for your credentials in chat to debug — never do it. The solution is always to look at the file directly or print a masked version. Credentials once pasted in chat should be considered exposed and regenerated immediately.

6 Supabase — Database Design

Why we use Supabase

Supabase gives us a hosted PostgreSQL database with a nice dashboard, auto-generated APIs, and a Python client library. We do not need to set up or manage a database server. The free tier is enough for development and early customers.

Table 1 — merchants

Stores each business that uses Vela. Every other table references merchant_id from this table.

CREATE TABLE merchants (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name        text NOT NULL,
  created_at  timestamptz DEFAULT now()
);

Table 2 — transactions

Records every payment attempt — whether it succeeds, fails, or is still waiting.

CREATE TABLE transactions (
  id                   uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  merchant_id          uuid REFERENCES merchants(id),
  checkout_request_id  text UNIQUE,
  mpesa_receipt        text UNIQUE,
  amount               numeric,
  phone                text,
  status               text DEFAULT 'pending',
  failure_reason       text,
  created_at           timestamptz DEFAULT now()
);
Why UNIQUE on checkout_request_id and mpesa_receipt? These are the idempotency guards. If Daraja sends the same callback twice (it sometimes does), the second attempt to insert will fail on the unique constraint rather than creating a duplicate paid record. This prevents double-crediting a payment.

Table 3 — daraja_logs

Logs every request sent to Daraja and every callback received from Daraja. This is the audit trail — if something goes wrong, you can replay what was sent and what came back.

CREATE TABLE daraja_logs (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  merchant_id uuid REFERENCES merchants(id),
  direction   text,   -- 'request', 'callback', 'callback_unmatched'
  payload     jsonb,
  created_at  timestamptz DEFAULT now()
);
Why jsonb? Daraja's request and callback payloads have different shapes. jsonb lets us store any structure without defining a rigid schema for every possible field Daraja might send. We can still query inside it.

Multi-tenancy: merchant_id on every table

Every table has a merchant_id column. This is the core of multi-tenancy. When merchant A's payment comes in, it is tagged with merchant A's ID at the moment of insert. When we query transactions for a dashboard, we always filter by merchant_id. This ensures one merchant can never see another's data — enforced at the database level, not just in application code.

How SQL runs in Supabase

In the Supabase dashboard, go to SQL Editor. You can open multiple query tabs. Running a CREATE TABLE in a new tab is safe — it does not affect any existing tables. Running an ALTER TABLE on an existing table modifies it in place without deleting its data.

7 How the Full Daraja STK Push Flow Works

Step 1 — Get OAuth Token
Before any request to Daraja, we must prove who we are. We send our Consumer Key + Consumer Secret (base64-encoded) to Daraja's OAuth endpoint. Daraja returns a time-limited access token (valid ~1 hour). We cache this token for 55 minutes so we do not hit Daraja on every single payment — that would slow things down and exceed rate limits.
Step 2 — Send STK Push request
We POST to Daraja's STK Push endpoint with: the shortcode, a base64 password (shortcode + passkey + timestamp), the customer's phone number, the amount, and our callback URL. Daraja validates our token and sends a payment prompt to the customer's phone. Daraja responds immediately with a CheckoutRequestID — this is the receipt number we use to track this transaction.
Step 3 — Customer acts
The customer sees a pop-up on their phone saying "Enter M-Pesa PIN to pay KES X to [business]". They either enter their PIN and pay, or they dismiss/ignore it (which counts as a failed transaction after timeout).
Step 4 — Daraja sends callback to Vela
After the customer acts (or times out), Safaricom's servers POST a JSON payload to our callback URL. The payload contains the same CheckoutRequestID, a result code (0 = success, anything else = failure), and on success: the M-Pesa receipt number and confirmed amount.
Step 5 — Vela updates the database
Our callback handler finds the pending transaction by CheckoutRequestID, checks it has not already been processed (idempotency), then updates the status to "paid" or "failed". The merchant's dashboard will reflect the real-time status.

app/daraja/auth.py — OAuth token with caching

Fetches an access token from Daraja. Caches it in memory for 55 minutes so we do not make a network request on every payment. If the cached token is still valid, it is returned immediately.

import os, base64, httpx
from datetime import datetime, timedelta
from dotenv import load_dotenv

load_dotenv()
CONSUMER_KEY = os.getenv("DARAJA_CONSUMER_KEY")
CONSUMER_SECRET = os.getenv("DARAJA_CONSUMER_SECRET")
ENVIRONMENT = os.getenv("DARAJA_ENVIRONMENT", "sandbox")
BASE_URL = "https://sandbox.safaricom.co.ke" if ENVIRONMENT == "sandbox" \
           else "https://api.safaricom.co.ke"

_cached_token = None
_token_expiry = None

def get_access_token() -> str:
    global _cached_token, _token_expiry
    if _cached_token and _token_expiry and datetime.now() < _token_expiry:
        return _cached_token
    credentials = f"{CONSUMER_KEY}:{CONSUMER_SECRET}"
    encoded_credentials = base64.b64encode(credentials.encode()).decode()
    response = httpx.get(
        f"{BASE_URL}/oauth/v1/generate?grant_type=client_credentials",
        headers={"Authorization": f"Basic {encoded_credentials}"},
    )
    response.raise_for_status()
    data = response.json()
    _cached_token = data["access_token"]
    _token_expiry = datetime.now() + timedelta(minutes=55)
    return _cached_token
Why 55 minutes? Daraja tokens expire after 60 minutes. Using 55 minutes gives a 5-minute safety buffer so a token cached near expiry does not get used and fail mid-request.

app/daraja/stk_push.py — Initiating the payment prompt

This function is called whenever we want to ask a customer to pay. It gets the token, builds the payload, logs the request to Supabase, sends the request to Daraja, and stores a pending transaction record.

import os, base64
from datetime import datetime
import httpx
from dotenv import load_dotenv
from app.daraja.auth import get_access_token, BASE_URL
from app.db.supabase import supabase

load_dotenv()
SHORTCODE = os.getenv("DARAJA_SHORTCODE")
PASSKEY = os.getenv("DARAJA_PASSKEY")

def initiate_stk_push(merchant_id: str, phone: str, amount: int,
                      account_reference: str, description: str = "Payment"):
    token = get_access_token()
    timestamp = datetime.now().strftime("%Y%m%d%H%M%S")
    password_str = f"{SHORTCODE}{PASSKEY}{timestamp}"
    password = base64.b64encode(password_str.encode()).decode()

    payload = {
        "BusinessShortCode": SHORTCODE,
        "Password": password,
        "Timestamp": timestamp,
        "TransactionType": "CustomerPayBillOnline",
        "Amount": amount,
        "PartyA": phone,
        "PartyB": SHORTCODE,
        "PhoneNumber": phone,
        "CallBackURL": os.getenv("DARAJA_CALLBACK_URL"),
        "AccountReference": account_reference,
        "TransactionDesc": description,
    }

    supabase.table("daraja_logs").insert({
        "merchant_id": merchant_id,
        "direction": "request",
        "payload": payload,
    }).execute()

    response = httpx.post(
        f"{BASE_URL}/mpesa/stkpush/v1/processrequest",
        json=payload,
        headers={"Authorization": f"Bearer {token}"},
    )
    response.raise_for_status()
    result = response.json()

    checkout_request_id = result.get("CheckoutRequestID")

    supabase.table("transactions").insert({
        "merchant_id": merchant_id,
        "checkout_request_id": checkout_request_id,
        "amount": amount,
        "phone": phone,
        "status": "pending",
    }).execute()

    return result
Why log the request BEFORE sending it? If Daraja returns an error, we still have a log of what we tried to send. If we logged after, a crash between sending and logging would leave a blind spot in the audit trail.

app/daraja/callbacks.py — Receiving payment confirmations from Daraja

Daraja calls this endpoint after a customer pays (or fails to pay). The handler finds the matching pending transaction, checks for duplicate callbacks (idempotency), and updates the status to "paid" or "failed".

from fastapi import APIRouter, Request
from app.db.supabase import supabase

router = APIRouter()

@router.post("/callbacks/stk-result")
async def stk_callback(request: Request):
    payload = await request.json()
    callback = payload["Body"]["stkCallback"]
    checkout_request_id = callback["CheckoutRequestID"]
    result_code = callback["ResultCode"]
    result_desc = callback["ResultDesc"]

    # Find the transaction we created when we sent the STK Push
    pending = (
        supabase.table("transactions")
        .select("id, merchant_id, status")
        .eq("checkout_request_id", checkout_request_id)
        .execute()
    )

    # If no matching transaction, log it and accept gracefully
    if not pending.data:
        supabase.table("daraja_logs").insert({
            "merchant_id": None,
            "direction": "callback_unmatched",
            "payload": payload,
        }).execute()
        return {"ResultCode": 0, "ResultDesc": "Accepted"}

    transaction = pending.data[0]
    merchant_id = transaction["merchant_id"]

    # Log every callback we receive
    supabase.table("daraja_logs").insert({
        "merchant_id": merchant_id,
        "direction": "callback",
        "payload": payload,
    }).execute()

    # Idempotency check — do not process if already marked paid
    if transaction["status"] == "paid":
        return {"ResultCode": 0, "ResultDesc": "Already processed"}

    # Handle failure (customer cancelled, wrong PIN, timeout)
    if result_code != 0:
        supabase.table("transactions").update({
            "status": "failed",
            "failure_reason": result_desc,
        }).eq("checkout_request_id", checkout_request_id).execute()
        return {"ResultCode": 0, "ResultDesc": "Accepted"}

    # Handle success
    items = callback["CallbackMetadata"]["Item"]
    metadata = {item["Name"]: item.get("Value") for item in items}  # .get() avoids KeyError on items with no Value
    supabase.table("transactions").update({
        "status": "paid",
        "mpesa_receipt": metadata.get("MpesaReceiptNumber"),
        "amount": metadata.get("Amount"),
    }).eq("checkout_request_id", checkout_request_id).execute()

    return {"ResultCode": 0, "ResultDesc": "Accepted"}
Why always return ResultCode 0? Daraja reads this response. If we return a non-zero code, Daraja treats it as an error on our end and may retry the callback — causing duplicate processing. Always return 0 to tell Daraja "we received it, do not retry."

9 ngrok — Making Daraja Reach Your Laptop

The problem ngrok solves

Your laptop runs on a private IP address (like 192.168.x.x). Daraja's servers on Safaricom's network cannot reach that address — it is behind your router's firewall and network address translation. Daraja needs a public URL to send the payment callback to. ngrok creates a secure tunnel from a public URL on the internet to your localhost:8000.

Running ngrok

# In a separate terminal (leave this running)
ngrok http 8000

ngrok prints a forwarding URL like https://spotless-treble-cure.ngrok-free.dev. That URL is what goes into your .env as DARAJA_CALLBACK_URL:

DARAJA_CALLBACK_URL=https://spotless-treble-cure.ngrok-free.dev/callbacks/stk-result
Why does the ngrok URL change? On the free plan, ngrok gives you a new random URL every time you restart it. When it changes, update the DARAJA_CALLBACK_URL in your .env file and restart uvicorn so the new URL is loaded. On the paid plan, you can reserve a static domain.

10 Errors Encountered — Causes and Fixes

Error 1 — "Attribute 'app' not found in module 'app.main'"
Symptom: uvicorn refused to start with this error even though app was clearly defined in main.py.

Root cause: Python caches compiled bytecode in __pycache__ folders. The cached version of app/main.py was outdated or corrupted — Python was reading the old cache instead of the actual file. The cache had no knowledge of the app object.
Fix: Delete the pycache folder and restart uvicorn.
Remove-Item -Recurse -Force app\__pycache__

After clearing the cache, Python recompiles from the actual source file and finds the app object correctly.
Error 2 — STK Push returning 404 Not Found
Symptom: The STK Push request was reaching Daraja's server (not a network issue) but getting a 404 response, as if the endpoint did not exist.

Root cause: A spelling typo in the HTTP headers dict: 
headers={"Authoriation": f"Bearer {token}"}
The word "Authorization" was misspelled as "Authoriation" (missing the 'z'). The header was silently ignored by httpx, so the request was sent without an Authorization header. Daraja rejected the unauthenticated request with 404 instead of 401 (Daraja's quirk).
Fix: Correct the spelling.
headers={"Authorization": f"Bearer {token}"}

Lesson: HTTP header names are case-sensitive strings. A single character typo in a string key is invisible to Python — the language has no way to know "Authoriation" is wrong. When you get a 404 from an API that should exist, check your headers before assuming the URL is wrong.
Error 3 — Indentation bug in stk_push.py
Symptom: The last few lines of stk_push.py ran when the file was imported, not when the function was called. Variables like checkout_request_id were undefined at module level, causing a NameError.

Root cause: The lines after response.raise_for_status() were accidentally indented at zero (module level) instead of four spaces (inside the function). Python uses indentation to define scope — module-level code runs once on import, function-level code runs when the function is called.
Fix: Indent all lines after response = httpx.post(...) to be inside the function (4 spaces). The return result line must also be inside the function.

Lesson: Python's indentation is not just style — it is the syntax that defines where a block of code belongs. An off-by-one indentation error moves code from inside a function to outside it with no warning.
Error 4 — ngrok "failed to fetch CRL" / SSL errors
Symptom: ngrok showed TLS/SSL errors when Daraja tried to reach the tunnel. Certificate Revocation List (CRL) fetch failures appeared in ngrok logs.

Root cause: Avast antivirus has a feature called "HTTPS Scanning" (under Scam Guardian → Web Guard). This feature intercepts all HTTPS connections and replaces SSL certificates with Avast's own certificates so it can inspect encrypted traffic. ngrok uses certificate pinning — it expects specific certificates — and Avast's substituted certificate caused the TLS handshake to fail.
Fix: In Avast, go to Menu → Settings → Protection → Scam Guardian → Web Guard. Uncheck "Enable HTTPS scanning". Restart ngrok.

ngrok then negotiates TLS directly without Avast intercepting it, and Daraja's callbacks reach the tunnel correctly.
Error 5 — Multiple zombie uvicorn processes all binding port 8000
Symptom: After repeated Ctrl+C and restarts, netstat -ano | findstr :8000 showed five different PIDs all listening on port 8000. The terminal being watched was not the process actually receiving requests. Logs from the watched terminal showed nothing even when requests arrived.

Root cause: uvicorn --reload spawns a parent process and a child process. When you press Ctrl+C, sometimes only one of them dies. Opening multiple VS Code terminal tabs and starting uvicorn in each one creates multiple server processes, all competing on port 8000. The operating system routes incoming requests to whichever process it chooses — not necessarily the one whose logs you are watching.
Fix: Open Task Manager → Details tab. Sort by "Image Name". Find all python.exe entries. Right-click each one → End Task. Then verify the port is clear:
netstat -ano | findstr :8000
It should return nothing. Then start one single clean uvicorn process in one terminal.

Why taskkill sometimes fails: If VS Code terminals are configured to auto-restart their process, killing a PID spawns a new one immediately. Killing from Task Manager (with End Task on the parent) is more reliable.
Error 6 — Callback returning 500 Internal Server Error ✓ RESOLVED
Symptom: Payment was deducted from the M-Pesa account. The ngrok inspector confirmed Daraja sent a callback to the tunnel. But the server returned 500 and the transaction stayed "pending" in Supabase.

Root cause — two layers:

Layer 1 (why we couldn't see the error): Five zombie uvicorn processes were all competing on port 8000. The terminal being watched was not the one actually receiving requests, so no traceback ever appeared. Once all processes were killed via Task Manager and a single clean server was started, the traceback became visible immediately.

Layer 2 (the actual bug) — KeyError: 'Value' in callbacks.py line 50:
Daraja's callback payload contains a list of items inside CallbackMetadata. Most items have both a "Name" and a "Value" key. But Daraja sometimes includes extra items that have only "Name" and no "Value" key at all. The code was using item["Value"] — Python's hard dictionary access — which crashes with a KeyError the moment a key does not exist.
Fix: Changed item["Value"] to item.get("Value") on line 50 of callbacks.py.

metadata = {item["Name"]: item.get("Value") for item in items}

Why .get() works: dict.get("key") is the safe version of dict["key"]. If the key does not exist, it returns None instead of crashing. The metadata dictionary is still built correctly from items that do have values, and the function completes without error.

Result: The full loop now works end to end — STK Push sent → customer pays → Daraja callback received → transaction updated to paid with M-Pesa receipt number in Supabase.

11 Key Concepts Learned Today

ConceptWhat it means in plain language
OAuth Token A temporary password Daraja issues to prove you are who you say you are. Expires in 60 minutes. Must be sent with every request.
STK Push Safaricom sends a payment pop-up to a customer's phone. The customer enters their PIN. Payment is complete without any physical interaction with the business.
Callback URL A public URL you give Daraja. After the customer acts, Safaricom's servers POST the result to this URL. Your code must be running and reachable at that URL.
CheckoutRequestID A unique ID Daraja returns when you send an STK Push. You store it. When the callback arrives, this ID links the callback back to your transaction record.
Idempotency Designing the system so that processing the same event twice produces the same result as processing it once. Achieved here via UNIQUE constraints on receipt numbers and status checks before updating.
Multi-tenancy One system serving many independent businesses, with data fully isolated between them. Achieved by putting merchant_id on every table and filtering by it in every query.
FastAPI Router A way to group related endpoints together in a separate file, then attach them to the main app. Keeps the code organized as the API grows.
Python __pycache__ Compiled bytecode cache Python maintains for speed. Can become stale and cause confusing errors. Safe to delete — Python regenerates it automatically.
Sandbox vs Production Sandbox: a testing environment where no real money moves. Uses fake shortcodes and credentials. Production: the live environment where real M-Pesa transactions happen.
dict["key"] vs dict.get("key") dict["key"] crashes with KeyError if the key is missing. dict.get("key") returns None safely. Always use .get() when a key might not exist — like in third-party API responses where you cannot control every field.
Zombie processes A process that is still running and holding a port after you thought you stopped it. On Windows with uvicorn --reload, Ctrl+C sometimes only kills the child. The parent keeps running, holds the port, and silently steals all incoming requests from any new server you start.

12 The POST /pay Endpoint

Why this endpoint matters

Before this, the only way to trigger an STK Push was to run a Python command in the terminal manually. That only works on a developer's laptop. The /pay endpoint turns the payment trigger into a proper API call — so the merchant dashboard, a WhatsApp bot, or any other client can trigger a payment by simply sending a POST request to /pay with a JSON body.

app/daraja/routes.py

from fastapi import APIRouter
from pydantic import BaseModel
from app.daraja.stk_push import initiate_stk_push

router = APIRouter()

class PayRequest(BaseModel):
    merchant_id: str
    phone: str
    amount: int
    account_reference: str
    description: str = "Payment"

@router.post("/pay")
def pay(request: PayRequest):
    result = initiate_stk_push(
        merchant_id=request.merchant_id,
        phone=request.phone,
        amount=request.amount,
        account_reference=request.account_reference,
        description=request.description,
    )
    return result
Why Pydantic BaseModel? FastAPI uses Pydantic to validate every incoming request automatically. If a caller sends a request missing phone or sends amount as a string instead of an integer, FastAPI rejects it with a clear 422 error before the code even runs. This is free input validation with zero extra code.

Registering the router in main.py

Every router must be registered in main.py using app.include_router(). Two routers now exist:

from app.daraja.callbacks import router as daraja_router
from app.daraja.routes import router as pay_router

app.include_router(daraja_router)
app.include_router(pay_router)
Why as pay_router? Both files export a variable called router. Without the alias, the second import would overwrite the first and one router would be lost. The as keyword gives each a unique name in the file.
